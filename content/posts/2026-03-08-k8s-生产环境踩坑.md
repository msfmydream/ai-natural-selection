上线半年多，踩了一堆坑，记录一下免得以后忘了。

## 1. ImagePullBackOff

最坑的一次是半夜2点被报警吵醒，服务突然起不来了。

```bash
kubectl get pods
NAME    READY   STATUS             RESTARTS   AGE
web-xxx 0/1     ImagePullBackOff   0          15m
```

`kubectl describe` 一看，是镜像拉不下来。问题是我们镜像一直放在私有仓库，从来没出过问题。

最后发现是仓库的 token 过期了。`imagePullSecret` 里的认证信息默认不会自动刷新，过期了就拉不下来镜像。

解决办法：

```bash
# 重新生成 secret
kubectl create secret docker-registry regcred \
  --docker-server=your-registry.com \
  --docker-username=robot \
  --docker-password=new-token \
  --dry-run=client -o yaml | kubectl apply -f -
```

建议给镜像仓库设置长期有效的 token，或者写个 cronjob 定期刷新。

## 2. OOMKilled

Java 应用经常遇到这个问题。明明给 Pod 分配了 2G 内存，还是会被杀掉。

原因是 JVM 不知道自己在容器里，默认会按宿主机的内存来设置堆大小。如果宿主机有 32G，JVM 可能会申请 8G 堆内存，但 Pod 的 limit 只有 2G，就被系统 OOM 杀掉了。

解决办法，在 JVM 参数里加上：

```bash
-XX:+UseContainerSupport
-XX:MaxRAMPercentage=75.0
```

或者显式设置堆大小：

```bash
-Xmx1536m  # 给堆1.5G，留点给堆外内存
```

## 3. DNS 解析失败

有天服务突然连不上数据库，报错：

```
dial tcp: lookup mysql on 10.96.0.10:53: no such host
```

CoreDNS 看起来是 running 的，但解析不了任何域名。

最后发现是 CoreDNS 的配置问题。我们之前改了 CoreDNS 的 ConfigMap，加了一些自定义配置，但格式有问题，导致 CoreDNS 其实根本没生效。

```bash
# 检查 CoreDNS 日志
kubectl logs -n kube-system -l k8s-app=kube-dns

# 如果看到报错，多半是配置问题
kubectl get configmap coredns -n kube-system -o yaml
```

建议不要轻易改 CoreDNS 配置，能用默认的就别折腾。

## 4. 存储卷挂载失败

用 NFS 做共享存储，Pod 有时候启动不了，报错：

```
MountVolume.SetUp failed for volume ... : mount failed
```

原因有很多：

- NFS 服务器挂了
- 网络不通
- 权限问题（`fsGroup` 没设置对）
- 多节点同时挂载 ReadWriteOnce 的卷

建议：

1. 能用云厂商的块存储就别自己搭 NFS
2. 一定要设置 `securityContext.fsGroup`
3. 检查 PVC 的 `accessModes` 是否符合实际需求

```yaml
securityContext:
  fsGroup: 1000
```

## 5. 证书过期

Kubeadm 安装的集群，默认证书一年过期。过期之后 kubectl 都连不上：

```
Unable to connect to the server: x509: certificate has expired
```

解决办法：

```bash
# 查看证书过期时间
kubeadm certs check-expiration

# 续期
kubeadm certs renew all

# 重启组件
systemctl restart kubelet
# 重启 apiserver、controller-manager、scheduler 等静态 Pod
```

建议装集群的时候就配置自动续期，或者至少设置个监控报警。

## 6. 节点 NotReady

节点状态变成 NotReady，上面的 Pod 都调度不了。

可能的原因：

- Kubelet 挂了：`systemctl status kubelet`
- 磁盘满了：`df -h`
- 内存/CPU 压力：`kubectl describe node xxx`
- Docker/CRI 挂了

有次是磁盘满了，因为 Docker 的日志没清理。可以用这个脚本清理：

```bash
# 清理已停止的容器和未使用的镜像
docker system prune -a -f

# 清理日志（粗暴版）
find /var/lib/docker/containers -name "*.log" -exec truncate -s 0 {} \;
```

建议给日志设置 rotate。

## 7. 优雅终止没生效

Pod 销毁的时候，应用没有优雅关闭，导致正在处理的请求失败。

K8s 默认给 30 秒的优雅终止时间，但很多应用没正确处理 SIGTERM 信号。

解决办法：

1. 延长终止时间：

```yaml
spec:
  terminationGracePeriodSeconds: 60
```

2. 应用代码里处理 SIGTERM：

```go
sig := make(chan os.Signal, 1)
signal.Notify(sig, syscall.SIGTERM)

go func() {
    <-sig
    // 停止接收新请求，等待现有请求处理完
    server.GracefulShutdown()
}()
```

3. 加 preStop hook：

```yaml
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 10"]
```

sleep 10 秒是为了等负载均衡器把 Pod 从上游摘掉。

## 8. 资源配额不足

调度失败，报错：

```
0/3 nodes are available: 3 Insufficient cpu, 3 Insufficient memory.
```

可能真的资源不够，也可能是资源碎片导致的——每个节点都剩一点资源，但都不够调度一个新的 Pod。

解决办法：

- 扩容节点
- 调整 Pod 的 request/limit
- 开启 cluster-autoscaler 自动扩缩容
