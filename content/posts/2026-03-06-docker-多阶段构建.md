上周被运维同事吐槽了，说我们项目的Docker镜像太大，部署一次要拉半天。

我看了看，确实有点离谱——一个Spring Boot应用，镜像居然1.2G。要知道这玩意儿打出来的jar包才80M啊，剩下的1.1G都是啥？

## 镜像里到底装了啥

用`docker history`一看，问题找到了：

```bash
$ docker history myapp:latest
IMAGE          CREATED        SIZE
abc123         2 days ago     1.2GB
<missing>      2 days ago     800MB   # openjdk:11 基础镜像
<missing>      2 days ago     200MB   # maven 构建依赖
<missing>      2 days ago     150MB   # 源码和构建缓存
```

基础镜像`openjdk:11`占了800M，构建时装的Maven占了200M，还有各种构建缓存、测试代码、甚至`.git`目录都在里面。

这哪是运行镜像，简直是把整个开发环境打包进去了。

## 多阶段构建：分离构建和运行

查了一圈资料，发现多阶段构建（multi-stage build）是解决这个问题的标准做法。

思路很简单：用一个镜像做构建（包含完整的JDK、Maven/Gradle），另一个镜像只跑应用（只需要JRE）。

这是优化后的Dockerfile：

```dockerfile
# 阶段1：构建
FROM maven:3.8-openjdk-11 AS builder

WORKDIR /build
COPY pom.xml .

# 先下载依赖，利用缓存层
RUN mvn dependency:go-offline -B

COPY src ./src
RUN mvn clean package -DskipTests

# 阶段2：运行
FROM openjdk:11-jre-slim

WORKDIR /app
COPY --from=builder /build/target/*.jar app.jar

EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

关键点是`COPY --from=builder`，只把构建产物（jar包）复制到运行镜像里，其他的全扔掉。

构建一下看看效果：

```bash
$ docker build -t myapp:optimized .
$ docker images | grep myapp
myapp            optimized      450MB
myapp            latest         1.2GB
```

从1.2G降到450M，效果立竿见影。

但还能再优化。

## 用 distroless 镜像更进一步

Google搞了一个叫distroless的镜像，里面连shell都没有，只有运行应用必需的文件。

```dockerfile
FROM gcr.io/distroless/java11-debian11

COPY --from=builder /build/target/*.jar app.jar

EXPOSE 8080
ENTRYPOINT ["java", "-jar", "/app.jar"]
```

构建出来的镜像只有120M。

不过这里有个坑——因为distroless镜像里没有shell，你没法`docker exec`进去调试。如果应用启动失败，排查问题会很麻烦。

我的做法是本地开发用普通的`openjdk:11-jre-slim`，生产环境才用distroless。

## 缓存优化：让构建更快

多阶段构建还有一个好处是可以优化缓存。

看这段：

```dockerfile
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
```

为什么要把`COPY pom.xml`和`COPY src`分开？

因为Docker是按层缓存的。如果`pom.xml`没变，即使`src`里的代码改了，`mvn dependency:go-offline`这一步也能直接用缓存，不用重新下载依赖。

在我们的CI环境里，这个优化把构建时间从8分钟降到了2分钟。

## 还能优化的点

1. **用 Alpine 基础镜像**：`openjdk:11-jre-alpine`比`openjdk:11-jre-slim`更小，但要注意Alpine用的是musl libc，有些应用可能会有兼容性问题。

2. **Spring Boot 分层打包**：Spring Boot 2.3+支持分层jar，可以把依赖层和应用层分开，只重新构建变化的部分。

3. **.dockerignore**：别忘了加这个文件，把`.git`、`target`、`.idea`这些没用的东西排除掉。

```
.git
.gitignore
README.md
Dockerfile
target
.idea
*.iml
```

## 最终效果

| 优化阶段 | 镜像大小 | 构建时间 |
|---------|---------|---------|
| 原始版本 | 1.2GB | 6分钟 |
| 多阶段构建 | 450MB | 4分钟 |
| 加缓存优化 | 450MB | 2分钟 |
| distroless | 120MB | 2分钟 |

现在部署速度快多了，运维同事也不吐槽了。

不过说实话，120M还是有点大。看看以后能不能上GraalVM搞 native image，据说能到几十M。
